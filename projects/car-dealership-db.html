Overview

The scenario was simple in description but rich in detail: build a database that captures a dealership's entire operation. Vehicles on the lot, customers walking in, orders being placed, payments tracked over time — all of it linked together so nothing falls out of sync.

I started with an entity-relationship diagram to map the world, then translated that into a normalized schema where each table had a clear purpose and foreign keys did the work of holding everything together.

Schema

  • Vehicle — vehicle name, vehicle ID, vehicle price.
  • Customer — first name, last name, address line 1, address line 2, city, postal code, phone number.
  • Order Info — order ID, customer ID, employee ID, vehicle ID, payment amount.
  • Payment History — payment ID, employee ID, customer ID.
  • Foreign keys linking customers to their orders, and orders to the vehicles and payments they involve.

Tech Stack

SQL RDBMS ER Diagram Schema Design Normalization

This project drilled home why data modeling matters before code does. A well-designed schema makes queries simple; a sloppy one makes everything painful. ER diagrams aren't optional — they're the blueprint.
